Day3-编写ORM

  • 在协程中，不能调用同步IO操作，因为所有用户都是由一个线程服务的，协程的执行速度必须非常快，才能处理大量用户的请求；若耗时的IO操作在协程中以同步的方式调用，则会等待一个IO操作时，系统无法响应任何其他用户；

  • 异步编程的一个原则：一旦决定使用异步，则系统每一层都必须是异步；

  • Web框架使用的基于asyncio的aiohttp，就是基于协程的异步模型；aiomysql为MySQL数据库提供了异步IO的驱动；

  • 创建一个全局的连接池，每个HTTP请求都可以从连接池中直接获取数据库连接，这样就不必频繁地打开和关闭数据库连接，而是能复用就尽量复用；

  • SQLite3的占位符是?，MySQL的占位符是%s；注意要始终坚持使用带参数的SQL，而不是自己拼接SQL字符串，这样可以防止SQL注入攻击；

  • yield from将调用一个子协程（也就是在一个协程中调用另一个协程）并直接获得子协程的返回结果；

  • async with ...为with语句添加的异步调用（将两个单词看成一个整体，含义类似于with await ...）；async for同理；

    • async with

Day4-编写Model

  • 对数据库授权命令（grant语句）

    grant select, insert, update, delete on awesome.* to 'www-data'@'localhost' identified by 'www-data'; # identified by后面为密码

  • Mysql中，float表示单精度浮点数，double或real表示双精度浮点数；

  • Mysql中key、index、primary key、unique key的区别：

    • key：主要用于加快查询速度的；可以简单地认为与index（索引）类似；
    • primary key：主键（键最多一个，但可以联合），值也是唯一的；
    • unique key：唯一键，只要求值是唯一的，但是键可以有多个；

Day5-编写Web框架

  • **inspect库用来获取类、函数的参数信息，类型检查等；**inspect.Parameter.kind有5种情况：

    • a. POSITIONAL_OR_KEYWORD：位置参数
    • b. VAR_POSITIONAL：可变参数，*key
    • c. VAR_KEYWORD：关键字参数，**key
    • d. KEYWORD_ONLY：命名关键字参数，*,key
    • e. POSITIONAL_ONLY：已过时，推荐使用VAR_POSITIONAL代替

  • 函数接收参数的优先级：先接收POSITIONAL_OR_KEYWORD，再接收KEYWORD_ONLY，最后接收VAR_POSITIONAL和VAR_KEYWORD（这两者没有交集）；(参考http://www.imooc.com/article/13053）

  • 为避免处理函数要分别从request中取相关关键数据，可以先统一预处理request中的关键数据，然后再传递给处理函数；
    每个处理函数定义的函数参数字段就是要预处理的数据；

    • a. request的关键数据来源：带参数的URL（比如/blog/{id}）、GET和POST的查询参数；这些都是从request对象中获取的，分别是dict(**request.match_info)、request.query_string等；

    • b. 处理函数可以定义使用位置参数、关键字参数、命名关键字参数去接收上面的关键数据；

    • c. 为了区分管理，这里使用位置参数去接收带参数的URL（比如/blog/{id}），使用命名关键字参数去接收GET和POST的查询参数，若不需要某种类型的数据就不定义对于类型的参数；（这只是一种方案）

    • d. 对于某些处理函数，可能还有必要将原始的request对象传入，以便获取其他相关信息；为了区分管理，这里规定使用最后一个位置参数来接收request；

    • e. 位置参数和命名关键字参数在函数传参时的区别在于：命名关键字参数必须指定参数名，而位置参数则是可有可无，其他没有任何区别；

    • f. 位置参数和命名关键字参数都只能接收定义的名称的参数，若有多余的参数传入调用则会报错；此时可以使用关键字参数去接收，其会将接收的所有参数包装在一个dict中传入；

    • g. 因为从request中获取的参数比较多，若处理参数没有定义关键字参数，则需要根据处理函数定义的参数名称，将多余的参数去除；

    • h. 位置参数和命名关键字参数调用时字段必须要有数据，否则报错；字段的数据可以是传入的，也可以是默认的；

    • 示例: 处理函数

def get_blog(id):
    pass

      @get('/api/comments')
def api_comments(*, page='1'):
    pass

  • 动态加载模块：__import__(module_name, globals(), locals())；

  • request.match_info：通过add_route注册的可变路径路由（如/{i}），request.match_info中会包含key为i，value为匹配的路径的信息；取值方式有两种，分别为dict(**request.match_info)、request.match_info.items()；参考http://aiohttp-zh.readthedocs.io/web.html#aiohttp-web-resource-and-route；

  • 定义RequestHandler：URL处理函数不一定是一个coroutine，因此可以自定义给一个RequestHandler类，用于从处理函数中分析其需要接收的参数，从request中获取必要的参数，调用URL函数，然后把结果转换为web.Response对象；

  • **middleware是一种拦截器，一个URL在被某个函数处理前，可以经过一系列的middleware的处理；**其可以改变URL的输入、输出，甚至可以决定不继续处理而直接返回；其主要用于把通用的功能从每个URL处理函数中拿出来，集中放到一个地方，比如一个一个记录URL日志的logger；

Day8-构建前端

  • 由于有多个前端页面，且每个页面都有相同的页眉和页脚；为了提高效率，可以使用通过include把页面拆成多个部分，比如

    <html>
    <% include file="inc_header.html" %>
    <% include file="index_body.html" %>
    <% include file="inc_footer.html" %>
</html>

  • 但是include方法不利于页面整体结构的维护，jinjia2的模板还可以通过“继承”方式实现模板的复用；

  • 编写一个“父模板”，定义一些可替换的block（块）；然后编写多个“子模板”，每个子模板都只需要替换父模板定义的block；比如：

    <!-- base.html -->
<html>
    <head>
        <title>{% block title%} 这里定义了一个名为title的block {% endblock %}</title>
    </head>
    <body>
        {% block content %} 这里定义了一个名为content的block {% endblock %}
    </body>
</html>

    <!-- 子模板a.html，只需要把父模板的title和content替换掉 -->

{% extends 'base.html' %}

{% block title %} A {% endblock %}

{% block content %}
    <h1>Chapter A</h1>
    <p>blablabla...</p>
{% endblock %}

    <!-- 子模板b.html -->

{% extends 'base.html' %}

{% block title %} B {% endblock %}

{% block content %}
    <h1>Chapter B</h1>
    <ul>
       <li>list 1</li>
       <li>list 2</li>
    </ul>
{% endblock %}

Day10-用户注册和登录

  • HTTP协议是一种无状态协议，而服务器要跟踪用户状态；

    • 使用直接从Session中取出用户登录信息的缺点是服务器需要在内存中维护一个映射表来存储用户登录信息，在多服务器上还需要对Session做集群；
    • 使用直接读取cookie的方式来验证用户登录，每次用户访问任意URL都对cookie进行验证；这种方式的好处是保证处理任意的URL都是无状态的，可以扩展到多台服务器；

  • 服务器可以使用下面算法构造防伪造cookie：

    "用户id" + "过期时间" + SHA1("用户id" + "用户口令" + "过期时间" + "SecretKey")
    • 当浏览器发送cookie到服务器端后，服务器收到：用户id、过期时间、SHA1值；
    • 若未到过期时间，服务器就根据用户id查找用户口令，并计算：SHA1("用户id" + "用户口令" + "过期时间" + "SecretKey")，之后并与浏览器cookie中的哈希进行比较，若相等则说明用户已登录，否则cookie就是伪造的；

    • jQuery.fn.extend(object)：给jQuery对象添加新的方法；
